# 一期：每日 AI 资讯

> 一期范围：用 **Python** 爬取指定站的「前一天」每日 AI 快讯，整理成纯文本，通过飞书群机器人 Webhook 推送。

---

## 需求

### 目标

从 [AI 工具集 - 每日 AI 快讯](https://ai-bot.cn/daily-ai-news/) 抓取「前一天」的快讯，整理成纯文本并推送至飞书群机器人。
![推送消息到飞书](https://i.mji.rip/2026/03/22/d3ad1e8bdf9e64c24fc75b4241e0a4dd.png)

### 数据来源

- **URL**：`https://ai-bot.cn/daily-ai-news/`
- **页面结构**：按日期分组（如「3月11·周三」），每组下有多条快讯；每条包含标题（带详情页链接）、摘要、来源（如「来源：机器之心」）。

### 功能需求

| 项 | 说明 |
|----|------|
| 发送时间 | 每天 **14:00** 由 cron 触发（Docker 一次性运行 `python -m src.main`）。 |
| 内容范围 | 取**前一天**对应日期分组下的所有条目，最多 **5 条**。 |
| 每条字段 | 标题、摘要、日期标签（如「3月11·周三」）、来源、**详情页 URL**（从列表页标题的 `href` 解析）。 |
| 输出 | 纯文本推送至飞书（环境变量 `FEISHU_WEBHOOK_URL`）；不写本地文件。 |
| 前一天缺失 | 若页面上没有「前一天」的日期分组，仍发送「昨日无数据」类提示到飞书。 |

### 非功能需求

- 爬取频率：仅配合定时任务每日一次。
- 部署：Docker 镜像 + `--env-file` 注入 `.env`，与 cotc-server-python 一致；服务器路径 `/usr/share/nginx/cotc-daily-news`。

### 后期扩展（规划）

- **其他资讯类型**：新增数据源与爬虫模块，在入口按类型或配置调度。
- **其他推送渠道**：企业微信、邮件等，可抽象为统一推送接口。

## 实现设计

下文说明整体架构、模块划分与关键流程，与 `README.md` 及上文需求保持一致。

### 总体架构

- **触发方式**：服务器 cron 每天 14:00 触发一次 Docker 容器运行：

  ```bash
  docker run --rm --env-file /usr/share/nginx/cotc-daily-news/.env cotc-daily-news:latest
  # 容器内执行等价于：python -m src.main
  ```

- **主流程（单次执行）**：
  1. 从环境变量中读取 `FEISHU_WEBHOOK_URL`。
  2. 计算「目标日期」= 当前日期 - 1 天（服务所在时区为 `Asia/Shanghai`）。
  3. 从 `https://ai-bot.cn/daily-ai-news/` 抓取页面，解析出该日期分组下的快讯列表，最多 5 条。
  4. 将快讯列表格式化为飞书要发送的纯文本内容：
     - 若有数据：按编号依次列出每条的标题、来源、摘要、详情页 URL 等。
     - 若无数据：生成「昨日无数据」类提示文案。
  5. 通过飞书自定义机器人 Webhook 将纯文本推送到指定群。

- **模块划分**：
  - `src/main.py`：应用入口与调度（编排）。
  - `src/ai_news/config.py`：AI 资讯相关配置（数据源 URL、条数限制等）。
  - `src/ai_news/crawler.py`：每日 AI 资讯爬取与解析。
  - `src/ai_news/formatter.py`：资讯列表到纯文本内容的格式化。
  - `src/common/feishu.py`：飞书 Webhook 发送封装。

---

### 目录与模块职责

#### `src/main.py`

**职责**：串联「日期计算 → 爬取 → 格式化 → 飞书推送」的主流程，作为 Docker 容器内的唯一入口。

**核心逻辑**：

- 读取环境变量：
  - 必需：`FEISHU_WEBHOOK_URL`。
  - 若缺失，可直接打印错误日志并退出（避免静默失败）。
- 日期处理：
  - 使用服务器本地时间（已配置为 `Asia/Shanghai`）。
  - `target_date = today - 1 day`，用于锁定「前一天」数据。
- 编排调用顺序：
  1. 调用 `ai_news.crawler.fetch_daily_news(target_date)` 获取最多 5 条资讯。
  2. 调用 `ai_news.formatter.format_news_list(news_list, target_date)` 生成文本内容。
  3. 调用 `common.feishu.send_text(webhook_url, content)` 发送至飞书。

#### `src/ai_news/config.py`

**职责**：集中管理与 AI 资讯爬取相关的配置常量，便于后续调整。

**配置项示例**：

- `BASE_URL = "https://ai-bot.cn/daily-ai-news/"`
- `MAX_ITEMS = 5`：单次推送的最大资讯条数。

后期如需调整数据源或条数限制，可在此文件修改，而不影响业务逻辑代码。

#### `src/ai_news/crawler.py`

**职责**：根据目标日期，从 `BASE_URL` 页面中找到对应日期分组，解析出资讯条目列表。

**输入与输出**：

- 输入：
  - `target_date: datetime.date`：表示要抓取的真实日期（例如 2026-03-10）。
- 输出：
  - `list[dict]`：资讯列表，每个元素为一个 `dict`，至少包含以下字段：
    - `title: str`：标题文本。
    - `summary: str`：摘要内容。
    - `date_label: str`：页面上的日期标签（如「3月11·周三」）。
    - `source: str`：来源（如「机器之心」）。
    - `url: str`：详情页完整 URL（从列表页标题 `href` 解析）。

**关键点**：

- **日期分组匹配**：
  - 页面按日期分组展示，例如「3月11·周三」。
  - 爬虫需将 `target_date` 转换为与页面文案一致的日期标签格式，或通过 HTML 结构定位到对应分组。
- **缺失处理（前一天无数据）**：
  - 若页面上找不到对应日期分组，函数返回空列表 `[]`。
  - 不在爬虫层面报错或中断，由上层统一处理「无数据」场景。
- **最大条数限制**：
  - 即使某日期分组下条目超过 5 条，也只取前 `MAX_ITEMS` 条，确保消息长度可控。

#### `src/ai_news/formatter.py`

**职责**：将爬取到的资讯列表转换为飞书要发送的纯文本内容，控制整体排版与文案。

**输入与输出**：

- 输入：
  - `news_list: list[dict]`：来自 `crawler` 的资讯列表。
  - `target_date: datetime.date` 或 `date_label: str`：用于在文案中显示日期。
- 输出：
  - `str`：最终发送给飞书的纯文本。

**建议格式**（示例思路，具体文案可在实现时微调）：

- 有数据时：

  ```text
  每日 AI 资讯 - 3月11·周三

  1. 【标题 A】（来源：机器之心）
  摘要：xxxxxx
  链接：https://example.com/a

  2. 【标题 B】（来源：量子位）
  摘要：yyyyyy
  链接：https://example.com/b

  （最多展示 5 条）
  ```

- 无数据时：

  ```text
  每日 AI 资讯

  昨日（3月11·周三）在「AI 工具集 - 每日 AI 快讯」未找到任何数据。
  ```

**逻辑要点**：

- 根据 `news_list` 是否为空，决定输出「正常列表」还是「无数据提示」。
- 对标题、来源、摘要、链接做基本的换行与缩进处理，保证在飞书中可读性较好。

#### `src/common/feishu.py`

**职责**：封装飞书自定义机器人 Webhook 的 HTTP 调用逻辑，对上层只暴露一个简单的发送接口。

**核心接口**（设计思路）：

- `send_text(webhook_url: str, content: str) -> None`
  - 负责将纯文本 `content` 发送到指定 `webhook_url`。
  - 内部使用 `requests.post`，构造飞书机器人要求的 JSON 结构。
  - 简单处理响应：
    - 成功时无需返回值，可按需打印调试信息。
    - 失败时打印错误码和响应内容，便于排查（不抛出未捕获异常，避免中断容器日志过于复杂）。

---

### 异常与边界情况处理

- **环境变量缺失**：
  - 若 `FEISHU_WEBHOOK_URL` 未配置：
    - 在 `main` 中检测并打印错误日志（例如 `FEISHU_WEBHOOK_URL is not set`）。
    - 直接返回，不进行后续爬取与推送。

- **网络请求失败**：
  - 爬取 `ai-bot.cn` 或调用飞书 Webhook 时，可能出现网络错误。
  - 建议在 `crawler` 和 `feishu` 模块中：
    - 捕获 `requests` 相关异常。
    - 打印简单错误信息。
    - 返回「空列表」或直接结束发送函数，避免抛出到最外层导致进程异常退出。

- **HTML 结构变更**：
  - 若 `ai-bot.cn` 页面结构调整，可能导致解析失败。
  - 可在 `crawler` 里适当增加健壮性判断：
    - 未找到预期节点时，视为「无数据」，返回空列表。
    - 在日志中标记「可能的 DOM 变更」提示。

### 扩展性考虑（为后期规划预留）

虽然一期只实现「每日 AI 资讯」，但本设计在以下方面为后续扩展预留了空间：

- **多数据源/多资讯类型**：
  - 当前的 `ai_news` 目录可以作为一个「资讯类型」的实现。
  - 若后续新增其他资讯类型（例如科技新闻、行业研究等），可以按照相同模式增加新的子目录（如 `tech_news/`），并在 `main` 中根据配置选择要执行的类型。

- **多推送渠道**：
  - 目前只提供 `common/feishu.py`。
  - 后续若增加企业微信、邮件等渠道，可以抽象出统一的「消息发送接口层」，例如 `common/notify.py`，在其中聚合不同实现。

通过上述设计，一期可以快速落地，同时保证未来演进时改动范围可控。
